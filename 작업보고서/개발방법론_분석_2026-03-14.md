# REVITA_LINK 개발 방법론 분석 보고서

**작성일**: 2026-03-14
**프로젝트**: REVITA_LINK
**목적**: BLE OTA 및 LoRa 통신 구현을 위한 최적 개발 방법 선정

## 1. 요구사항 분석

### 1.1 필수 기능

| 기능 | 설명 | 난이도 |
|------|------|--------|
| **BLE OTA** | 무선 펌웨어 업데이트 | ★★★★★ |
| **LoRa 통신** | LoRaWAN 또는 Star(1:N) | ★★★★☆ |
| **주변장치 제어** | MCP23017, 모터, LED, 부저 등 | ★★☆☆☆ |

### 1.2 현재 상태

- **플랫폼**: PlatformIO + Nordic nRF52 (v10.11.0)
- **보드**: nRF52840 (RAK4631 호환)
- **현재 코드**: Bare-metal (SDK 미사용)
- **검증 완료**: 2.4GHz Radio (Nordic Proprietary), LoRa (SX1262 직접 제어)

### 1.3 핵심 과제

```
┌─────────────────────────────────────────────────────────┐
│  BLE OTA 구현                                           │
│  ├── Bootloader 필요 (MCUboot, Nordic DFU 등)          │
│  ├── BLE Stack 필요 (SoftDevice, Zephyr BLE 등)        │
│  └── DFU Service 구현 필요                              │
│                                                         │
│  LoRa 통신 구현                                         │
│  ├── Option A: LoRaWAN (복잡, 표준 호환)               │
│  └── Option B: Star Topology (단순, 자체 프로토콜)      │
└─────────────────────────────────────────────────────────┘
```

## 2. 개발 방법론 비교

### 2.1 Option 1: Bare-Metal 계속 (현재 방식)

```
현재 코드 기반 확장
├── BLE: 직접 구현 (RADIO 레지스터)
├── OTA: 직접 구현 (Bootloader + Flash 관리)
└── LoRa: 현재 코드 확장
```

| 항목 | 평가 |
|------|------|
| BLE OTA 구현 | ❌ **매우 어려움** - BLE 프로토콜 스택 전체 구현 필요 |
| LoRa 구현 | ✅ 가능 - 현재 코드 확장 |
| 개발 기간 | ❌ 6개월+ (BLE OTA만) |
| 유지보수 | ❌ 어려움 |
| 코드 크기 | ✅ 최소 |

**결론**: BLE OTA를 bare-metal로 구현하는 것은 **비현실적**

---

### 2.2 Option 2: Nordic nRF5 SDK + SoftDevice

```
nRF5 SDK (v17.1.0)
├── SoftDevice S140 (BLE 5.0 Stack)
├── BLE DFU (Buttonless OTA)
├── Flash Storage (FDS)
└── 사용자 코드
    ├── LoRa 드라이버 (현재 코드 이식)
    └── 주변장치 드라이버
```

| 항목 | 평가 |
|------|------|
| BLE OTA 구현 | ✅ **SDK 제공** - Buttonless DFU 예제 있음 |
| LoRa 구현 | ⚠️ 수동 통합 필요 - SDK에 없음 |
| 개발 기간 | 🔶 2-3주 (학습 포함) |
| 유지보수 | ✅ Nordic 공식 지원 |
| 코드 크기 | 🔶 중간 (~150KB with SoftDevice) |

**장점**:
- BLE OTA가 이미 구현되어 있음 (ble_app_buttonless_dfu 예제)
- 안정적인 BLE 스택 (SoftDevice)
- 풍부한 예제 코드

**단점**:
- SoftDevice가 Radio를 독점 → 시간 분할 필요 (Timeslot API)
- LoRaWAN 스택 별도 통합 필요
- SDK 구조 학습 필요

---

### 2.3 Option 3: Zephyr RTOS

```
Zephyr RTOS
├── BLE Stack (Host + Controller)
├── MCUboot (Bootloader)
├── LoRaWAN Stack (내장)
├── Device Drivers
└── 사용자 코드
    └── Application Logic
```

| 항목 | 평가 |
|------|------|
| BLE OTA 구현 | ✅ **MCUboot + SMP** 지원 |
| LoRaWAN 구현 | ✅ **내장 지원** (LoRaMac-node 기반) |
| 개발 기간 | 🔶 3-4주 (학습 포함) |
| 유지보수 | ✅ 오픈소스, 활발한 커뮤니티 |
| 코드 크기 | ❌ 큼 (~200KB+) |

**장점**:
- BLE OTA와 LoRaWAN 모두 내장
- RTOS 기능 (멀티스레딩, 타이머 등)
- Nordic, RAK 공식 지원
- 향후 확장성 좋음

**단점**:
- 학습 곡선 가파름
- 빌드 시스템 복잡 (west, CMake)
- 메모리 사용량 큼
- 기존 bare-metal 코드 전면 재작성 필요

## 3. LoRa 통신 방식 선택

### 3.1 LoRaWAN vs Star Topology

| 항목 | LoRaWAN | Star (1:N) |
|------|---------|------------|
| **복잡도** | ★★★★★ | ★★☆☆☆ |
| **구현 기간** | 4-6주 | 1-2주 |
| **표준 호환** | ✅ 국제 표준 | ❌ 자체 프로토콜 |
| **게이트웨이** | 필요 (TTN, ChirpStack 등) | 자체 Hub 구현 |
| **확장성** | ✅ 수천 노드 | 🔶 수십 노드 |
| **보안** | ✅ AES-128 내장 | 별도 구현 필요 |
| **전력 소비** | Class A 최적화 | 직접 최적화 필요 |

### 3.2 권장 LoRa 방식

**REVITA_LINK 특성 분석**:
- 예상 노드 수: 수십 대 (소규모)
- 환경: 실내/제한된 범위
- 데이터: 센서 값, 제어 명령 (소량)

```
권장: Star Topology (1:N) 우선 구현
      → 추후 LoRaWAN 업그레이드 고려
```

**이유**:
1. 초기 개발 속도 빠름
2. 자체 인프라로 운영 가능
3. 소규모 네트워크에 적합
4. 향후 LoRaWAN 마이그레이션 가능

## 5. 기술적 세부사항

### 5.1 BLE OTA 구현 (nRF5 SDK)

```
Bootloader 구조:
┌─────────────────────┐ 0x00000000
│  MBR (4KB)          │
├─────────────────────┤ 0x00001000
│  SoftDevice (152KB) │
├─────────────────────┤ 0x00027000
│  Application        │
├─────────────────────┤
│  DFU 영역           │
├─────────────────────┤
│  Bootloader (24KB)  │
└─────────────────────┘ 0x00100000

Flash 레이아웃 (1MB):
- MBR:        0x00000000 - 0x00001000 (4KB)
- SoftDevice: 0x00001000 - 0x00027000 (152KB)
- App:        0x00027000 - 0x000D8000 (708KB)
- Bootloader: 0x000F4000 - 0x000FE000 (40KB)
- Settings:   0x000FE000 - 0x00100000 (8KB)
```

**필요 컴포넌트**:
```
nRF5_SDK_17.1.0/
├── components/
│   ├── ble/ble_services/ble_dfu/
│   ├── libraries/bootloader/
│   └── softdevice/s140/
└── examples/
    └── dfu/
        ├── secure_bootloader/
        └── experimental/buttonless_dfu/
```

### 5.2 Star Topology 프로토콜 설계 (안)

```
패킷 구조:
┌────┬────┬──────┬────────┬─────┐
│TYPE│SRC │ DST  │PAYLOAD │ CRC │
│1B  │2B  │ 2B   │ N Bytes│ 2B  │
└────┴────┴──────┴────────┴─────┘

TYPE:
  0x01: BEACON (Hub → All)
  0x02: DATA (Node → Hub)
  0x03: ACK (Hub → Node)
  0x04: CMD (Hub → Node)
  0x05: CMD_ACK (Node → Hub)

통신 시퀀스:
  Hub                          Node
   │                             │
   │────── BEACON ──────────────>│ (주기적)
   │                             │
   │<─────── DATA ───────────────│ (센서값)
   │                             │
   │──────── ACK ───────────────>│
   │                             │
   │──────── CMD ───────────────>│ (제어명령)
   │                             │
   │<────── CMD_ACK ─────────────│
   │                             │

타이밍:
- Hub BEACON: 5초마다
- Node 전송 윈도우: BEACON 후 (NodeID × 100ms)
- 재전송: 3회 시도, 500ms 간격
```

### 5.3 Timeslot API (BLE + LoRa 공존)

```c
// SoftDevice Timeslot 요청 예시
static nrf_radio_request_t m_timeslot_request = {
    .request_type = NRF_RADIO_REQ_TYPE_EARLIEST,
    .params.earliest = {
        .hfclk = NRF_RADIO_HFCLK_CFG_XTAL_GUARANTEED,
        .priority = NRF_RADIO_PRIORITY_NORMAL,
        .length_us = 5000,  // 5ms LoRa 작업
        .timeout_us = 50000
    }
};

// Timeslot에서 LoRa 작업 수행
void timeslot_handler(void) {
    // SoftDevice가 Radio 해제한 동안
    // SX1262 TX/RX 수행
    sx1262_transmit(packet);
}
```
