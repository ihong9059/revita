### 3. 주요 작업 내용

#### 3.1 개발환경 및 RTOS 필요성 분석

기존 드라이버 테스트 코드와 FEATURE_LIST.md를 전수 분석하여 다음을 정리:

1. **현재 개발환경**: nRF52840 (RAK4631), Zephyr RTOS (NCS), west 빌드, J-Link SWD
2. **검증 완료된 Zephyr 기능 10개**: 커널 타이머, UART IRQ, GPIO, ADC, LoRa, USB CDC, QSPI, I2C, Devicetree, printk
3. **Zephyr 의존 분석**: LoRa/USB CDC/QSPI는 Zephyr 드라이버 기반 → bare-metal 전환 시 전면 재작성 필요
4. **동시성 요구 분석**:
   - Tower: 4+ 스레드 (LoRa RX / RS485 폴링 / LTE MQTT / USB CDC)
   - Link: 3+ 스레드 (LoRa RX / RS485 센서 / 모터 타이머)
5. **결론**: Zephyr RTOS 필수 (기존 의존성, 동시성, 타이밍 제약, 생태계, 전환 비용)

### 7. 핵심 파악 사항

1. **단일 루프 한계**: 현재 드라이버 테스트는 `main()` while(1) 루프로 충분하지만, 통합 펌웨어에서는 LoRa ACK 1.5s 데드라인 vs LTE AT 180s 타임아웃 등 타이밍 충돌이 불가피

2. **Zephyr 락인은 이점**: LoRa subsys, USB CDC, QSPI, MCUboot 등 이미 Zephyr 생태계에 깊이 의존 중이며, 이는 전환 비용이 아니라 활용 가능한 자산

3. **스레드 구조가 다음 설계 핵심**: Tower 4개, Link 3개 스레드 + ISR 구조를 먼저 확정해야 통합 펌웨어 작성 가능
