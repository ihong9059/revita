### 3. 주요 작업 내용

#### 3.1 Node(Link) 전체 기능 반영 — 웹 UI 1차

Node의 전체 기능을 웹 UI에 표현:

- **Output**: LED ON/OFF, Buzzer ON/OFF, Motor X CW/STOP/CCW, Motor Y CW/STOP/CCW
- **Input - Sensor**: 토양센서 #1/#2 (온도/습도), 엽온센서 (온도/습도)
- **Input - Status**: Button, Vibration, Flow ×2 (pulse count), Battery (프로그레스 바)

Tower 펌웨어에 `CMD:MOTOR_X_CW/CCW/STOP`, `CMD:MOTOR_Y_CW/CCW/STOP`, `CMD:SENSOR_READ`, `CMD:INPUT_READ` 추가.

#### 3.2 LTE 세션 프로토콜 설계

REVITA 시스템의 핵심 특성 반영:
- **Tower가 주기적으로 LTE 연결 → 데이터 교환 → 해제** (상시 연결 아님)
- **명령은 큐에 적재 → 다음 세션에 일괄 전달** (즉시 push 불가)

10단계 세션 흐름 정의:
1. CONNECT (LTE attach)
2. SESSION_START (Tower 상태 보고)
3. SESSION_ACK (Host 수락 + 대기 명령 수)
4. TELEMETRY_BATCH (센서 데이터 일괄 업로드)
5. UPLOAD_ACK (수신 확인 → QSPI 레코드 삭제)
6. CMD_REQUEST (명령 요청)
7. CMD_BATCH (Host → Tower 명령 전달)
8. CMD_ACK (수신 확인)
9. SCHEDULE (다음 연결 주기 협상)
10. SESSION_END (연결 해제)

스케줄링: Normal(30min), Frequent(5min), Power-save(2hr), On-demand

#### 3.3 Host Web — 세션 기반 전면 재작성

app.py:
- `execute_session()`: 세션 실행 엔진 (텔레메트리 수신 → 명령 전달 → 기록)
- `queue_command()` / `cancel_command()`: 명령 큐 관리
- 텔레메트리 스냅샷: 마지막 세션 데이터 유지
- 세션 히스토리, 스케줄 설정 API

index.html — 2패널 레이아웃:
- 좌측: Sensors, Status/Counter, Output Control (큐 방식)
- 우측: Command Queue, Session History, Schedule, Comm Log, Connection
- Session Banner: 연결 상태, 마지막 세션 시각, 대기 명령 수, Run Session 버튼
- 토스트 알림: "Queued: MOTOR_X_CW"

### 7. 핵심 파악 사항

1. **LTE 주기적 연결 = 명령 큐 패턴 필수**: 상시 연결이 아니므로 Host에서 명령을 큐에 쌓고, Tower 세션 시 일괄 전달하는 구조가 필요

2. **세션 = 하나의 트랜잭션**: Tower가 연결 → 데이터 업로드 + 명령 다운로드 → 해제까지가 하나의 단위. UPLOAD_ACK 전에는 QSPI 레코드 삭제 금지

3. **스케줄 협상**: Host가 다음 연결 주기를 동적으로 조절 가능 (관수 중 5분 / 평상시 30분 / 절전 2시간)

4. **즉시 연결 조건**: 배터리 위험, 진동 감지, 유량 급변 등은 스케줄 무시하고 즉시 연결
