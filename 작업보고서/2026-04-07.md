#### 3.7 Node RS485 wire 송신 (system/driver/rs485_test/node)

- uart0 (P0.20/P0.19) @ 9600 8N1
- DE=P1.04, RE#=P1.03, 12V_EN=P0.17 (모두 직접 GPIO)
- 매 2초 `ND-RS485:n\r\n` 송신
- 디버깅 이력:
  1. **break만 보임** → cutecom이 115200 → 9600 으로 수정
  2. **`\r\n`이 `<0xff>`로 깨짐** → DE→RX 전환 전 `k_busy_wait(2000)` 으로 shift register flush 확보
- **사용자 wire 검증 완료** ✅

#### 3.8 Tower RS485 wire 송신 — 미해결

- uart0 (P0.20/P0.19) @ 115200, console과 wire 공유
- DE=P0.17, RE#=P0.21
- MCP23017 @ 0x20 → MUX_SEL=LOW, MUX_EN#=LOW, 3V3/5V/12V_EN=HIGH
- 디버깅 이력 (다수):
  1. UART 매핑 오류 (uart1 → LTE 핀) → uart0로 정정
  2. 9600 가정 → kc_cert/tower의 115200으로 정정
  3. 12V/3V3/5V_EN 추가/제거 반복
  4. DE/RE# 영구 HIGH 시도
  5. MCP 초기화 순서 변경
  6. **Buzzer test 진행 중 발견**: I2C0 핀이 명시적으로 잡혀있지 않아 MCP가 처음부터 응답 안 했을 가능성
     - 수정: overlay에 i2c0 핀 P0.13(SDA)/P0.14(SCL) 명시 + 100kHz 로 클럭 낮춤
     - rs485_test/gateway에도 동일 수정 적용
- **결과**: ttyUSB1 디버그는 정상, RS485 A/B wire는 여전히 미확인 → HW 의심
- 코드는 kc_cert/tower 패턴으로 정리해 마무리, HW 검증은 다음 세션

#### 3.11 Buzzer (system/driver/buzzer_test/{tower,node}) + Tower MCP 진단

- **Node** (Link): 12V_EN=P0.17, BUZZER_EN=P0.24 — 모두 직접 GPIO
- **Tower**: 12V_EN=MCP23017 GPA2, BUZZER_EN=MCP23017 GPB1
- 패턴: 100ms beep × 5회 (2초 간격) → 종료
- **Tower 첫 시도 → MCP init FAIL (rc=-2)**: 0x20-0x27 에서 응답 없음
- 진단 추가:
  - i2c0 pinctrl을 명시적으로 P0.13(SDA)/P0.14(SCL) 지정
  - 클럭 100kHz로 낮춤
  - 0x08~0x77 전체 스캔
- **결과**: Tower MCP 정상 응답, 양쪽 buzzer 동작 ✅
- **이는 큰 진전**: kc_cert/system/tower의 RS485가 의심되는 이유와 같은 원인일 가능성

### 7. 핵심 파악 사항

1. **Tower vs Link 디버그 UART 핀 매핑이 다름**:
   - Tower: `&uart0` + P0.20/P0.19 (RAK_UART1) → ttyUSB1
   - Link: `&uart1` + P0.16/P0.15 (RAK_UART2) → ttyUSB0
   - kc_cert/system/link 의 과거 overlay 는 RX=P0.02 (잘못된 핀)였으나 TX 만 사용했기에 발견 안 됐었음
   - Node RX 검증을 위해 console_rx_thread 패턴을 정립 (uart_poll_in 폴링)

2. **Tower I2C0 (MCP23017) 핀 명시 필요**:
   - 기본 board overlay 만으로는 MCP 가 응답 안 함 (rc=-2)
   - 해결: overlay 에 `i2c0_default_revita { TWIM_SDA P0.13, TWIM_SCL P0.14 }` 추가
   - 클럭은 400kHz → 100kHz (Standard) 로 낮춤
   - **이전 RS485 실패의 원인이 같은 곳에 있었을 가능성** — 이전엔 MCP 자체가 안 잡혀서 MUX 활성 코드가 무용지물이었던 것
   - rs485_test/gateway 에 동일 수정 적용했지만 wire 는 여전히 미확인 (HW 의심)

3. **9600 baud 에서 RS485 TX→RX 전환 시 shift register flush 필수**:
   - `uart_poll_out` 은 ENDTX 까지만 기다리고 shift register 는 보장하지 않음
   - 9600 baud 에서 1 byte ≈ 1.04 ms
   - 마지막 byte 의 stop bit 까지 보장하려면 `k_busy_wait(2000)` (2 ms) 필요
   - Node RS485 에서 200us → 2000us 변경 후 정상 동작

4. **엽온 센서가 습도도 측정함**:
   - RS485 부록 PDF 분석 결과 "엽온"이지만 실제론 잎의 온도 + 습도 동시 측정
   - 텔레메트리 구조 재배치 필요 (`leaf_humid_pct` 추가, `flow_count u16 → flow_delta u8`)

5. **Tower MUX 라우팅 → 디버그/RS485가 같은 UART 핀**:
   - P0.20 (RAK_UART1) → TMUX1574 → RS485 트랜시버
   - 외부 USB-UART 디버그 어댑터도 P0.20 에 직접 와이어로 연결됨 (사용자 환경)
   - 따라서 디버그 console 과 RS485 wire 가 같은 UART 를 공유
   - DE=LOW 일 때는 트랜시버 high-Z → wire 에 안 나감 → 디버그만 보임

6. **LoRa SF12 운용 검증됨**:
   - RSSI -46~-65 dBm, SNR +5~+8
   - 안테나 약함 이슈는 SF12 로 충분히 보상됨
